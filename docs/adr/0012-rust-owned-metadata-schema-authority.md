# Rust-owned metadata schema is the validating authority

The `ls-metadata` serde structs and validator are the single authority that defines and gates per-TR maintenance metadata: closed-set facets are modelled as Rust enums (so an unknown `owner_class`, `protocol`, `rate_bucket`, `instrument_domain`, `venue_session`, or `certification_path` is a located parse error), and `validate_dir` enforces that every indexed TR has a per-TR file and that `tr-index.yaml`'s routing fields equal the per-TR values. We chose this over a hand-maintained JSON Schema because the runtime already needs these types in Rust (the `{TR}_POLICY` consts in `ls-core` mirror the index, and a dev-test cross-checks them), so a parallel JSON Schema would be a second source of truth to keep in sync for no added safety. Any JSON Schema we ship later is generated from these Rust types when hand-authoring friction appears — never hand-edited (plan R7).
